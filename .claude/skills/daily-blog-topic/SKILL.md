# 오늘의 블로그 글감 찾기 (daily-blog-topic)

주제가 정해지지 않은 "오늘 뭐 쓸까" 요청부터, 인스타 URL이나 브랜드 소식 하나만 던져진 요청까지 — 글감을 확정하고 발행까지 끝낸다.

이 스킬은 프로젝트 루트의 `CLAUDE.md`를 전제로 한다(신발 데이터 구조, 카테고리 분류 기준, 신상 프로토콜, 가격 데이터 원칙, 리뷰 작성 가이드라인 등은 전부 거기 있다 — 여기서 반복하지 않는다). 충돌 시 `CLAUDE.md`가 우선이다. 신발을 정면으로 비교하는 글이 이미 확정됐다면 이 스킬이 아니라 `shoe-comparison-post`로 간다.

## 시작 전 파악

- `date +%Y-%m-%d`로 오늘 날짜를 확인한다. 시의성 판단(오늘 마감/오늘 출시 여부)의 기준점이다.
- 사용자가 이미 구체적인 소스(인스타 URL, 브랜드/모델명, 뉴스 링크)를 줬는가? 줬으면 백로그 훑기 없이 바로 "2. 신선한 트리거 탐색"으로 간다.
- **용어가 애매하면(브랜드 캠페인 신조어, 해시태그처럼 들리는 표현, 축약어) 조사를 시작하기 전에 먼저 "어디서 봤어요? 링크 있어요?"부터 물어라.** 브랜드가 방금 만든 용어는 기존 지식으로 추론이 안 되고, 잘못 짚으면 세션 하나를 통째로 날린다 — 실제로 "팀우먼스"를 후원 엘리트 선수단으로 짐작하고 조사했다가, 나중에 인스타 URL을 받고서야 실제로는 여성 러너 모집 프로그램(NB Team Women's)이었음을 확인한 사례가 있다. 소스가 명확하면 이 단계는 생략한다.

## 1. 백로그·발행 이력 확인 (사용자가 소스를 안 준 경우)

- 세션 메모리 시스템에 글감 백로그 성격의 파일이 있는지 확인한다(보통 이름에 `content-backlog`가 들어간다; 메모리 인덱스 파일에서 찾는다). 없으면 이 프로젝트에 아직 그런 메모리가 없다는 뜻이니 건너뛴다.
- `git log --oneline -20`으로 최근 발행 흐름을 파악한다.
- **백로그에 "미발행"이라 적힌 항목은 stale할 위험이 크다.** 반드시 `grep -r "slug: '<후보>'" src/lib/data/blog/posts/`로 실제 미발행인지 재확인한 뒤 진행한다 — 과거 세션에서 백로그의 "미발행" 다수가 이미 발행 완료 상태였던 사례가 여러 번 있었다.

## 2. 신선한 트리거 탐색

- **사용자 제공 소스 우선.** URL이나 모델명이 있으면 그것부터 조사한다.
- **인스타그램**: 특정 게시물 URL 하나는 **WebFetch로 로그인 없이 캡션 전문을 바로 확인**할 수 있다(og:description에 캡션이 담긴다). 반면 계정 프로필을 스캔해서 새 게시물을 찾는 작업은 로그인 세션이 필요해 Chrome MCP(사용자의 로그인된 세션)로만 가능하다 — researcher 서브에이전트 위임은 로그인 벽 때문에 실패한다. 모니터링 대상 계정 목록과 우선순위는 `CLAUDE.md`의 "모니터링 대상 계정" 표를 따른다.
- **신상 신발 출시 임박/당일**: 한국 출시 여부·정가·RunRepeat 랩 데이터 게시 여부를 재확인한다. "글로벌은 오늘 출시인데 한국은 아직 조용하다"는 사실 자체가 훅이 될 수 있다(예: 호카 클리프톤 프로 — 한국 어디에도 리스팅이 없다는 점이 기사의 각도가 됨). 신상 데이터 추정 처리는 `CLAUDE.md`의 "신상 신발 대응 프로토콜"을 따른다.
- **마라톤/대회 접수 시즌**: 접수 임박이라고 무조건 새 글을 쓰지 않는다. 기존에 그 대회를 다룬 글이 있으면 신규 발행보다 `updatedAt` 갱신이 정답인 경우가 많다(자기잠식 회피).
- **브랜드 공식 발표**: 런칭 캘린더, newsroom(`CLAUDE.md`의 "브랜드별 미디어/프레스 페이지" 표)도 트리거 소스다.

## 3. 사실 검증 원칙 — WebSearch 요약을 그대로 믿지 않는다

WebSearch 결과 요약은 여러 출처를 뭉뚱그려 서술해서, 그럴듯하지만 특정 원문에는 없는 문장을 만들어내는 경우가 있다. **구체적이고 검증 가능한 사실 주장(정확한 스펙 수치, "이 신발이 이 선수의 발 데이터로 설계됐다" 같은 구체적 인과 관계 주장)은 WebFetch로 원문 1차 소스를 직접 대조한다.** 오늘 세션에서 실제로 이 검증을 생략했다면 근거 없는 서사를 그대로 발행글에 넣을 뻔했다 — WebSearch 요약과 원문(BITR)·다른 원문(DOR)이 서로 다른 얘기를 하고 있었다.

신발 DB의 기존 값(특히 와이드 옵션, 가격, 플레이트 유무)도 맹신하지 않는다 — 공식 소스로 교차검증한다.

## 4. 중복·자기잠식 확인

`grep`으로 같은 주제·모델이 이미 다뤄졌는지 확인한다. 겹치면 각도를 재설정하거나(다른 상대와 비교, 다른 관점), 접수 갱신류면 기존 글 업데이트로 전환한다. 애매하면 사용자에게 확인한다.

## 5. 주제 확정

여러 각도가 가능하면 근거와 함께 2~3개 옵션을 제시하고 사용자에게 확인받는다(추측으로 밀어붙이지 않는다). 신상이라 DB 추가 여부가 갈리면(한국 정가 확정 여부에 따라) 그 판단도 여기서 사용자와 공유한다.

## 6. 작성

`src/lib/data/blog/posts/{발행월 YYYY-MM}.ts`의 배열(`posts_YYYY_MM`) **맨 앞**에 새 객체로 Edit 삽입한다. (해당 월 파일이 없으면 생성 후 `posts/index.ts`에 import+스프레드 등록. 2026-07부터 월별 분할됨)

- 카테고리(`news`/`guide`/`tips`/`review`) 판단 기준은 `CLAUDE.md`의 "카테고리 자동 판단 기준" 표를 따른다.
- 메타: `id`/`slug`(소문자·하이픈), `author: '산초 에디터'`, `publishedAt`은 오늘, `readingTime`은 본문 분량에 맞게, `tags`.
- 본문 구조는 기존 유사 카테고리 글 1~2편을 먼저 Read해서 스타일을 맞춘다. 공통적으로 쓰이는 검증된 구조: `tldr`(3줄 요약) → `callout info`(맥락) → 필요하면 `h2` 스펙테이블 → 본문 `h2` 섹션들 → `callout warning`(추정·미확정 사항 명시) → `FAQ`(`div class="faq"`) → `<p><small>` 출처 각주. 스타일이 정의 안 된 임의 클래스(`verdict` 등)는 쓰지 않는다.
- **본문의 모든 `/shoes/{slug}`·`/blog/{slug}` 내부 링크는 삽입 전에 `grep`으로 실존을 확인한다.** 죽은 링크는 SEO·신뢰 모두 깎는다.
- 이미 다른 글에서 "확정되면 별도로 다루겠다"고 예고해둔 게 있으면, 그 예고 문장을 찾아 새 글로 링크를 연결해 약속을 이행한다(`grep`으로 관련 문구 검색).

## 7. 썸네일

기본 경로: **SVG(1200×630) + sharp composite**.

1. 프로젝트 루트에 `scratch-thumb.cjs` 같은 임시 스크립트를 작성한다(Pretendard 폰트, 사이트 컬러 sky/slate/navy + 카테고리별 accent — news는 보라/핑크, review는 스카이블루 계열처럼 톤을 구분해도 좋다).
2. 신발 이미지가 있으면 `public/images/shoes/{brand}/{model}/side.webp`를 회전시켜 SVG 배경에 합성한다. **`sharp().rotate()`에 `{ background: { r: 0, g: 0, b: 0, alpha: 0 } }`를 반드시 지정한다** — 안 하면 회전으로 생긴 여백이 검은 사각형으로 채워진다.
3. 실행 후 `Read`로 결과 이미지를 육안 확인한다.
4. 작업이 끝나면 스크래치 스크립트를 삭제한다(레포에 남기지 않는다).

Chrome 확장이 연결돼 있고 여러 신발을 카드형으로 나란히 배치해야 하는 경우(예: 3파전 비교), `shoe-comparison-post` 스킬의 base64 인라인 캡처 방식(`.claude/skills/shoe-comparison-post/references/thumbnail.md` 참고)이 대안이다.

## 8. 검증 체인

- 신발 데이터(`shoes/*.ts`)를 건드렸으면 `npm run validate`(에러 0 확인, 경고는 통과).
- `npm run build`(Compiled successfully + 새 글의 정적 페이지 생성 확인).
- **빌드가 통과하지 못하면 완료로 보고하지 않는다.**

## 9. 커밋·배포

- 빌드가 재생성한 로컬 sitemap은 되돌린다: `git checkout public/sitemap-0.xml` — 로컬 빌드 시점엔 방금 만든 커밋이 아직 git 로그에 없어 `next-sitemap`이 산출하는 lastmod가 부정확할 수 있고, 어차피 Vercel이 배포 시 정확히 재생성한다.
- `git add`: `src/lib/data/blog/posts/` + 썸네일 webp + (신발 데이터를 건드렸으면 해당 `shoes/*.ts`).
- 커밋 메시지: `feat(blog): {요약}` + 본문에 각도·시의성 근거.
- **push 타이밍**: 기본은 한 세션에 여러 편(3~4편)을 개별 커밋으로 쌓고 **마지막에 한 번만 push**한다(배포 노이즈·롤백 리스크 관리). **예외는 진짜 시계에 묶인 시의성뿐이다** — 오늘 마감인 모집 공고, 오늘이 정확히 글로벌 출시일인 신상처럼 "미루면 내용 자체가 틀려지거나 의미가 없어지는" 경우만 즉시 개별 push한다. 애매하면(예: "출시 당일"이 훅이지만 `publishedAt` 날짜만 맞으면 되는 경우) 사용자에게 지금 push할지 물어본다.

## 10. 후속 기록

- 신상 추정치가 들어간 경우 `.omc/todo-estimates.md`(없으면 생성)에 재확인 트리거를 기록한다(무엇을·언제 재확인할지).
- 세션 메모리의 content-backlog 파일(있다면)에 이번 발행 내역과 각도·교훈을 추가한다.
- 이번 세션에서 새로 확인된 작동 패턴이나 실수(예: 특정 소스의 신뢰도 문제, 새로운 추출 방법)가 있으면 관련 메모리 파일도 함께 갱신한다.

## 완료 보고

선정한 각도(왜 이걸 골랐는지, 다른 후보는 왜 제외했는지), slug, 검증 결과(build 통과 여부), 배포 여부와 그 판단 근거(즉시 push했다면 왜 예외였는지)를 간결하게 요약한다.
